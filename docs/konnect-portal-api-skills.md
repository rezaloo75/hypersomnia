# Konnect Portal & API Builder — Agent Skills Reference

This document describes the Konnect v3 Dev Portal and API Builder APIs as they actually behave in production. It is intended for:

- **AI coding agents** building integrations against the Konnect Portal APIs
- **Documentation teams** writing guides for the Konnect Dev Portal

All response shapes are confirmed from live production data, not inferred from documentation alone.

---

## Authentication

All Konnect API calls require a Personal Access Token (PAT):

```
Authorization: Bearer <PAT>
```

Regions: `us`, `eu`, `au`, `global`. Most Portal and API Builder resources use the `global` or `us` region.

---

## API Versioning: v2 vs v3

Konnect has two active API generations. **They are not interchangeable.**

| Version | Used for |
|---|---|
| `v2` | Control planes, services, routes, plugins, legacy portals |
| `v3` | API Builder (APIs, implementations, publications), Dev Portal (portals, applications, developers) |

**Critical**: Portal IDs generated by the v3 system will return `404 "portal not found"` if passed to v2 portal endpoints. Always use `/v3/portals/...` when working with portals created in the current Konnect UI.

---

## Pagination

Two different pagination styles depending on API version:

### v2 — offset-based
```
GET /v2/control-planes?size=100&offset=<cursor>
Response: { data: [...], offset: "<next_cursor>" }
```

### v3 — cursor-based
```
GET /v3/apis?page[size]=100&page[after]=<cursor>
Response: { data: [...], meta: { next: { cursor: "<next_cursor>" } } }
```

Advance pages while `meta.next.cursor` (v3) or `offset` (v2) is present.

### List API Implementations
Links an API in the API Builder to a Kong service on a control plane.
```
GET /v3/api-implementations?page[size]=100
```
```json
{
  "data": [
    {
      "id": "...",
      "api_id": "2e01e704-9689-4aaa-8e11-b0d50f662339",
      "service": {
        "id": "c47b1d61-c6b2-4790-9c04-25802110f7b0",
        "control_plane_id": "71b3562b-84ec-4bf4-a280-2490255dcd25"
      }
    }
  ]
}
```

To find which API(s) back a given Kong service:
```ts
const matchedApiIds = implementations
  .filter(impl =>
    impl.service?.id === serviceId &&
    impl.service?.control_plane_id === cpId
  )
  .map(impl => impl.api_id)
```

### List Portal Applications
```
GET /v3/portals/{portalId}/applications?page[size]=100
```

> ⚠️ Use `/v3/portals/...`, not `/v2/portals/...`. v3 portal IDs will 404 on the v2 endpoint.

**Confirmed production response shape:**
```json
{
  "data": [
    {
      "id": "6a44c43c-6cbc-43e0-ab67-98ed276d19cb",
      "name": "FlightCast",
      "description": "Alex's app",
      "status": "approved",
      "created_at": "2026-02-23T01:11:07.496Z",
      "updated_at": "2026-02-23T01:11:07.496Z",
      "labels": {},
      "registration_count": 2,
      "portal": { "id": "160bc48c-c151-439b-882f-35c36c066bb3" },
      "owner": {
        "id": "54cfd935-7cf2-4b66-9bea-2e0c0c6dd9d6",
        "type": "developer"
      },
      "auth_strategy": {
        "id": "62541ad3-db78-421c-bea7-77810016fdfa",
        "name": "key-auth",
        "credential_type": "key_auth",
        "key_names": ["apikey"]
      }
    }
  ],
  "meta": { "page": { "total": 1 } }
}
```

**Key facts:**
- The application object does **not** embed developer/owner name — only `owner.id`.
- To resolve the owner's name, make a separate call to the developers endpoint (see below).
- `registration_count` is a summary count; full registration details require a separate call.

### Get Developer (resolve owner name)

The application's `owner.id` is a developer ID. Fetch the developer to get their name:

```
GET /v3/portals/{portalId}/developers/{developerId}
```
```json
{
  "id": "54cfd935-7cf2-4b66-9bea-2e0c0c6dd9d6",
  "full_name": "Reza Shafii",
  "email": "reza@example.com",
  "status": "active"
}
```

Resolve display name in order: `full_name ?? name ?? email`

### List Application Registrations

Fetch which API versions an application is registered for:

```
GET /v3/portals/{portalId}/applications/{appId}/registrations?page[size]=100
```
```json
{
  "data": [
    {
      "id": "...",
      "status": "approved",
      "created_at": "...",
      "api": {
        "id": "2e01e704-9689-4aaa-8e11-b0d50f662339",
        "name": "AeroAPI"
      },
      "api_product_version": {
        "id": "...",
        "name": "v1"
      }
    }
  ]
}
```

**Matching a registration to a specific API** (field name varies; check all three):
```ts
function registrationMatchesApi(reg, apiId, apiVersionMap) {
  if (reg.api?.id === apiId) return true
  if (reg.api_product?.id === apiId) return true
  const vId = reg.api_product_version?.id
  return vId != null && apiVersionMap.has(vId)
}
```

## Common Data-Flow Pattern: Route → Portal Applications

Given a Kong **route** (with a backing **service** on a **control plane**), find all Dev Portal applications registered for the API that backs it:

```
1. GET /v3/api-implementations
      filter: impl.service.id === serviceId
              AND impl.service.control_plane_id === cpId
      result: matchedApiIds[]

2. GET /v3/api-publications
      filter: pub.api_id ∈ matchedApiIds
      result: matchedPortalIds[]

3. For each portalId in matchedPortalIds:
      GET /v3/portals/{portalId}/applications
      result: apps[]

4. For each app in apps (parallel):
      GET /v3/portals/{portalId}/applications/{appId}/registrations
      GET /v3/portals/{portalId}/developers/{app.owner.id}

5. Filter apps: keep only those with ≥1 registration matching a matchedApiId

6. Display: app name, owner name, registration date, status, API name
```

All steps 3 and 4 can be parallelized with `Promise.all`.

## Known Gotchas

| Issue | Detail |
|---|---|
| v2/v3 portal ID mismatch | `/v2/portals/{id}/applications` returns 404 for v3 portal IDs. Always use `/v3/portals/{id}/applications`. |
| No owner name on application | `GET /v3/portals/{id}/applications` returns `owner.id` only. Name requires `GET /v3/portals/{id}/developers/{ownerId}`. |
| Registration field name varies | The API field on a registration may be `api`, `api_product`, or resolvable only via the version ID. Check all three. |
| Portal domain missing | Some portals may have neither `custom_domain` nor `default_domain` set — handle gracefully (skip or show ID only). |
| Pagination style differs | v2 uses `size`/`offset`; v3 uses `page[size]`/`page[after]`. Mixing them silently returns wrong pages. |
